# 方案 A 核心功能设计方案 (第一阶段)

## 1. 系统架构概述
围绕 `TradingAgents-CN` 现有功能构建轻量级“盯盘解读”闭环，重点在于**自选股卡片流监控与可配置的定时解读**。
- **前端 (Vue 3)**: 提供完全响应式的监控中心。卡片流展示自选股最新行情、风险评分、AI 解读摘要。包含“查看详情”跳转，并支持配置监控规则（盘前/盘中/盘后）。
- **后端 (FastAPI)**:
    - 引入 `Digest` 和 `WatchRule` 领域实体。
    - `watch_rules` 集合：存储自选股的解读频率配置（如“交易日 15:00 执行”）。
    - `digests` 集合：存储执行完生成的摘要结果。
    - **定时任务引擎 (APScheduler)**: 后台轮询配置并触发分析。
- **底层分析引擎**: 复用 `TradingAgents-CN` 分析流获取结论和风险标签，从中提取精简版的 summary 返回并存储为 `Digest`。

## 2. 数据流与存储 (MongoDB)
1. **用户配置**: 存储在 `watch_rules` 表。包含 `user_id, stock_code, cron_expr/schedule_type, status`。
2. **分析结果富集**: `digests` 表记录 `stock_code, user_id, report_id, summary, risk_score, created_at`。
3. **接口扩展**: 
    - `/api/favorites` 中引入富集选项或新开 `/api/watch/digests` 聚合接口，输出适用于卡片展示的合并数据。
    - 接口设计应**兼顾后续小程序的极简请求需求**。

## 3. 核心界面设计 (纯 Web 响应式 - 卡片流)
- **页面布局**:
    - **Header/Filter**: 搜索框、分类标签（风险等级、板块）、手动“一键全量解读”按钮。
    - **Content Area**: 响应式 Grid 布局（PC 3-4 列，移动端单列）。
- **监控卡片 (Watch Card) 元素**:
    - 顶部：股票名称、代码、最新价格及涨跌幅（红/绿颜色区分）。
    - 中部：最新的 AI 摘要文字（支持高亮关键词）。
    - 底部：风险/机会标签、更新时间、解读配置状态（如“每天盘后”）。
    - 操作区：【配置策略】、【刷新解读】、【完整报告】。

## 4. 后台调度任务 (Scheduler)
- 利用现有的 `scheduler_service`，动态挂载用户的定时解读任务。
- 解读任务调用 `create_analysis_task`，通过监听任务状态获取最终结果。
- 生成的完整报告存入 `analysis_reports`，提取的轻量级摘要写入 `digests` 并可通过 SSE 或 WebSocket 推送到前台。

## 5. 错误处理与容错
- **LLM 限流与成本控制**: 如果同一时刻并发任务过多，加入队列和优先级（现有 `QueueService`），避免 API 超载。
- **数据降级**: 如果分析任务失败，卡片上显示上一次的摘要，并在界面标注“更新失败”，附带重试按钮。
